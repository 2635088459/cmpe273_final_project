End-to-End Workflow

Request lifecycle

Every deletion request moves through a well-defined state machine with step-level visibility.

Request Status States
Per-Service Step States
PENDING RUNNING SUCCEEDED FAILED RETRYING SKIPPED_CIRCUIT_OPEN

Each service reports its own step status back via RabbitMQ. The orchestrator aggregates into a final request state.

PARTIAL_COMPLETED is intentional

If non-critical services (e.g. analytics) fail but primary + cache + search succeed, the system reports partial completion — allowing auditors to see exactly which systems were cleaned.

Reliability Design

Built for failure, not against it

Three layered mechanisms ensure every deletion eventually completes — or is explicitly accounted for.

🔁

Retry + Dead-Letter Queue

  • Failed messages re-queue with exponential backoff
  • After N attempts, routed to DLQ for manual inspection
  • DLQ replay endpoint lets ops re-process without data loss
  • RabbitMQ TTL + x-dead-letter-exchange wiring
🛡️

Idempotency Guards

  • processed_events table with unique (event_id, service_name)
  • PostgreSQL unique constraint raises 23505 on duplicate → service skips silently
  • Duplicate events logged to proof chain as DUPLICATE_EVENT_IGNORED
  • Safe to re-deliver messages at any time
⚡

Circuit Breaker

  • Per-service circuit stored in Redis
  • CLOSED → OPEN after failure threshold
  • OPEN steps report SKIPPED_CIRCUIT_OPEN — no wasted retries
  • Half-open probe automatically resets after cooldown
  • Admin UI shows real-time breaker state per service

Proof & Auditability

Tamper-evident audit chain

Every deletion event is hash-chained. Modify any record and verification fails — cryptographically.

🔗
Hash-chained proof eventsEach event records previous_hash + event_hash (SHA-256). Any tampering breaks the chain.
🔒
Advisory lock during appendPostgreSQL pg_advisory_xact_lock ensures sequential per-request chain writes under concurrent load.
📤
Export & verifyFull proof chain exportable as JSON. One-click "Verify Chain" traverses all hashes and reports integrity status.
📋
Compliance-readyDeletion evidence satisfies GDPR Art. 17 and CCPA deletion audit requirements out of the box.

Consistency & Tradeoffs

Eventual consistency, intentionally

Not all systems need the same consistency guarantee. We made deliberate tradeoffs per service.

System Strategy Latency Consistency Rationale
PostgreSQL Synchronous Low Strong Source of truth; must be consistent first
Cache (Redis) Async Event Very Low Eventual TTL provides natural expiry as backup
Search Index Async Event Low Eventual Search staleness acceptable for seconds
Analytics Soft-delete / Delayed High (deferred) Eventual Historical batch jobs tolerate delay; immutable logs preferred
Backup Scheduled High (scheduled) Eventual Backup rotation cycle handles deletion on next run
⚖️

Why PARTIAL_COMPLETED is good

Rather than failing the whole request if analytics is slow, we record partial success. Auditors see exactly which systems were cleaned — more informative than a binary pass/fail.

📐

CAP Theorem position

We favour Availability + Partition Tolerance (AP) for the event-driven path. The proof chain provides Consistency through eventual convergence and tamper detection.
